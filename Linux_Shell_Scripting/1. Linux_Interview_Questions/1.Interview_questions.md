## Process Management

### 1. An application is stuck / unresponsive. How do you find and kill it?
**Trigger Point:** Tests basic process troubleshooting under pressure.
**Command:**
```bash
ps aux | grep <process_name>
kill -9 <PID>
```
**Flags Explained:**
- `ps aux` → `a` = show processes for all users, `u` = user-oriented format (shows CPU/MEM), `x` = include processes without a controlling terminal.
- `kill -9` → sends `SIGKILL`, force-terminates immediately (no cleanup). Prefer `kill -15` (`SIGTERM`) first to allow graceful shutdown.

---

### 2. How do you find which process is consuming the most CPU/memory in real time?
**Trigger Point:** Server is slow — needs live triage.
**Command:**
```bash
top
htop
ps aux --sort=-%cpu | head -10
ps aux --sort=-%mem | head -10
```
**Flags Explained:**
- `--sort=-%cpu` → sorts descending by CPU%; `-%mem` sorts by memory.
- `top` interactive keys: `Shift+P` (sort by CPU), `Shift+M` (sort by memory), `k` (kill from within top).

---

### 3. A zombie process is showing up. What does it mean and how do you handle it?
**Trigger Point:** Tests understanding of process states.
**Command:**
```bash
ps aux | grep 'Z'
ps -o pid,ppid,state,cmd -C <process_name>
kill -HUP <parent_PID>
```
**Flags Explained:**
- Zombie (`Z` state) = process finished execution but its exit status hasn't been read by the parent. You can't kill a zombie directly — you kill/restart the **parent process**, which lets init/systemd reap it.
- `-o` → custom output columns.

---

### 4. How do you find which process is holding a file open (preventing deletion/unmount)?
**Trigger Point:** "Device is busy" or you can't delete a log file even after deleting it.
**Command:**
```bash
lsof | grep <filename>
lsof /mount/point
fuser -v <filename>
```
**Flags Explained:**
- `lsof` = list open files. `-v` in `fuser` gives verbose output showing PID, user, and access mode.
- Common real scenario: disk shows 100% full via `df` but `du` doesn't match — a deleted file is still held open by a process; find PID via `lsof | grep deleted` and restart that process to reclaim space.

---

### 5. How do you run a long process that survives SSH session disconnect?
**Trigger Point:** Deploying/running scripts over SSH from a laptop that may lose connection.
**Command:**
```bash
nohup ./script.sh &
disown
# or
screen -S session_name
tmux new -s session_name
```
**Flags Explained:**
- `nohup` → ignores `SIGHUP` sent when terminal closes.
- `&` → runs in background.
- `disown` → removes job from shell's job table so it's not killed on logout.
- `screen`/`tmux` → creates a detachable terminal multiplexer session (preferred in real infra work since you can reattach with `screen -r` / `tmux attach`).

---

### 6. How do you check what a process is doing / which files and ports it's using?
**Trigger Point:** Debugging a mystery process before restarting it in production.
**Command:**
```bash
ls -l /proc/<PID>/fd
cat /proc/<PID>/status
lsof -p <PID>
```
**Flags Explained:**
- `/proc/<PID>/fd` → shows open file descriptors as symlinks (files, sockets, pipes).
- `lsof -p` → all open files for a specific PID.

---

### 7. How do you change the priority (niceness) of a running process?
**Trigger Point:** A backup job is hogging CPU and starving the app.
**Command:**
```bash
nice -n 10 ./script.sh
renice -n 5 -p <PID>
```
**Flags Explained:**
- `nice` range: -20 (highest priority) to 19 (lowest). Default is 0.
- `renice` changes priority of an already-running process; requires root to lower niceness (increase priority) of another user's process.

## Disk & Storage

### 8. Disk shows 100% full via `df` but `du` says there's free space. Why, and how do you fix it?
**Trigger Point:** Classic "phantom disk full" scenario — very commonly asked.
**Command:**
```bash
df -h
du -sh /* 2>/dev/null | sort -rh | head -10
lsof +L1
```
**Flags Explained:**
- `df -h` → disk usage at filesystem level, human-readable.
- `du -sh` → summarized, human-readable size per directory; `sort -rh` sorts reverse-human-readable (biggest first).
- Root cause: deleted files still held open by a running process consume space until the process closes the file handle. `lsof +L1` finds files with link count < 1 (i.e., deleted but open).
- Fix: restart the process holding the file, or `> /proc/<PID>/fd/<fd_number>` to truncate it live.

---

### 9. How do you find the top 10 largest files/directories under a path?
**Trigger Point:** Disk cleanup before it hits 100%.
**Command:**
```bash
du -ahx / | sort -rh | head -10
find / -xdev -type f -size +100M -exec ls -lh {} \;
```
**Flags Explained:**
- `-a` → include files, not just directories; `-x` → stay on one filesystem (don't cross into mounted volumes).
- `find -xdev` → same purpose, avoids scanning NFS/other mounts.
- `-size +100M` → files larger than 100MB.

---

### 10. How do you check disk I/O and identify which process is causing high disk wait?
**Trigger Point:** High load average but CPU% looks normal → likely I/O bound.
**Command:**
```bash
iostat -xz 1
iotop -oa
vmstat 1
```
**Flags Explained:**
- `iostat -x` → extended stats (await, %util per device); `-z` → skip devices with zero activity.
- `iotop -o` → show only processes doing I/O; `-a` → accumulated (not per-second) I/O.
- `vmstat` → `wa` column shows % CPU waiting on I/O; high `wa` confirms disk bottleneck.

---

### 11. A partition/LVM volume is running out of space. How do you extend it (without downtime)?
**Trigger Point:** Production disk growth — common in EC2/on-prem VM scenarios.
**Command:**
```bash
lsblk
growpart /dev/xvda 1
resize2fs /dev/xvda1        # ext4
xfs_growfs /                # xfs
```
**Flags Explained:**
- `lsblk` → view block devices/partitions.
- `growpart` → extends the partition to use newly added disk space (after resizing the underlying EBS/disk).
- `resize2fs` resizes ext-family filesystem; `xfs_growfs` resizes XFS (used by default in RHEL/Amazon Linux) — note XFS **can only grow, never shrink**.

---

### 12. How do you check filesystem type, mount points, and mount a new disk?
**Trigger Point:** Attaching new EBS volume in AWS.
**Command:**
```bash
lsblk -f
mkfs -t xfs /dev/xvdf
mkdir /data
mount /dev/xvdf /data
blkid
```
**Flags Explained:**
- `lsblk -f` → shows filesystem type and UUID per device.
- `mkfs -t xfs` → format device with specified filesystem type.
- To persist across reboot, add an entry to `/etc/fstab` using the UUID from `blkid` (not `/dev/xvdf`, since device names can shift).

---

### 13. How do you check inode usage, and what happens if inodes are exhausted?
**Trigger Point:** "No space left on device" even though `df -h` shows free space.
**Command:**
```bash
df -i
find / -xdev -printf '%h\n' | sort | uniq -c | sort -rn | head
```
**Flags Explained:**
- `df -i` → shows inode usage %, separate from block/space usage.
- Cause: millions of tiny files (e.g., session files, logs) exhaust inodes even with disk space free. Fix = delete unnecessary small files, not just free space.

## File Permissions & Ownership

### 14. A script fails with "Permission denied" even though the user owns the file. What do you check?
**Trigger Point:** Deployment scripts failing in CI/CD pipelines.
**Command:**
```bash
ls -l script.sh
chmod +x script.sh
stat script.sh
namei -l /full/path/to/script.sh
```
**Flags Explained:**
- `chmod +x` → adds execute permission.
- `namei -l` → walks the full path and shows permissions at every level (parent directories need execute/traverse permission too — a very common gotcha).

---

### 15. Explain `chmod 755` vs `chmod 644` — what do the numbers mean?
**Trigger Point:** Fundamental permissions question, asked to test real understanding vs memorization.
**Command:**
```bash
chmod 755 script.sh
chmod 644 file.txt
```
**Flags Explained:**
- Permission digit = sum of read(4) + write(2) + execute(1), per **owner / group / others**.
- `755` = owner: rwx(7), group: r-x(5), others: r-x(5) — typical for scripts/executables.
- `644` = owner: rw-(6), group: r--(4), others: r--(4) — typical for regular files/configs.

---

### 16. What's the difference between `chmod`, `chown`, and `chgrp`? And what does the setuid/setgid/sticky bit do?
**Trigger Point:** Tests depth beyond basic file permission commands.
**Command:**
```bash
chown user:group file
chgrp group file
chmod u+s /usr/bin/passwd     # setuid
chmod g+s /shared/dir         # setgid
chmod +t /tmp                 # sticky bit
```
**Flags Explained:**
- `chown` changes owner (and optionally group with `user:group`); `chgrp` changes group only.
- **setuid (`u+s`)**: executable runs with the file owner's privileges (e.g. `passwd` runs as root even when a normal user executes it).
- **setgid (`g+s`)** on a directory: new files inherit the directory's group (used for shared team directories).
- **Sticky bit (`+t`)** on a directory: only the file owner can delete their own files, even if others have write access (used on `/tmp`).

---

### 17. How do you recursively change ownership/permissions for a deployment directory?
**Trigger Point:** App fails to start because app user can't read config/log directories.
**Command:**
```bash
chown -R appuser:appgroup /opt/app
find /opt/app -type d -exec chmod 755 {} \;
find /opt/app -type f -exec chmod 644 {} \;
```
**Flags Explained:**
- `-R` → recursive.
- Using `find -type d/-type f` separately (instead of `chmod -R 755`) avoids incorrectly giving execute permission to regular files.

---

### 18. How do you find all files with world-writable or SUID permissions (security audit scenario)?
**Trigger Point:** Security/compliance audit question.
**Command:**
```bash
find / -xdev -type f -perm -0002 -ls
find / -xdev -perm -4000 -ls
```
**Flags Explained:**
- `-perm -0002` → world-writable files (security risk).
- `-perm -4000` → files with the SUID bit set (potential privilege escalation vector — should be audited/minimized).

---

## Networking

### 19. A service isn't reachable on its port. How do you troubleshoot step by step?
**Trigger Point:** One of the most common scenario questions — tests structured troubleshooting.
**Command:**
```bash
systemctl status <service>
ss -tulnp | grep <port>
netstat -tulnp | grep <port>
curl -v localhost:<port>
telnet <host> <port>
firewall-cmd --list-all      # or: iptables -L -n
```
**Flags Explained:**
- `ss -tulnp` → `t`=TCP, `u`=UDP, `l`=listening sockets, `n`=numeric (no DNS lookup), `p`=show owning process. (`ss` is the modern replacement for `netstat`.)
- Troubleshooting order: is the process running? → is it listening on the expected port/interface? → is a local firewall blocking it? → is it reachable from another host?

---

### 20. How do you check network connectivity and diagnose packet loss/latency to a remote host?
**Trigger Point:** Intermittent connectivity issues between services.
**Command:**
```bash
ping -c 4 <host>
traceroute <host>
mtr <host>
```
**Flags Explained:**
- `-c 4` → send exactly 4 packets and stop.
- `traceroute` shows the network hops; `mtr` combines ping + traceroute continuously, useful for spotting where packet loss occurs.

---

### 21. How do you check which process is listening on port 8080, and how do you free it?
**Trigger Point:** "Address already in use" when starting a service.
**Command:**
```bash
lsof -i :8080
ss -ltnp | grep 8080
kill -9 <PID>
```
**Flags Explained:**
- `lsof -i :8080` → shows the exact process bound to port 8080.
- Always confirm the correct process before killing — in production, prefer stopping the service properly via `systemctl stop` if it's a managed service.

---

### 22. How do you check DNS resolution issues?
**Trigger Point:** Service can't resolve an internal/external hostname.
**Command:**
```bash
nslookup example.com
dig example.com +short
cat /etc/resolv.conf
getent hosts example.com
```
**Flags Explained:**
- `dig +short` → concise output, just the resolved IP.
- `getent hosts` → resolves using the same resolution path the OS/applications use (respects `/etc/nsswitch.conf`, `/etc/hosts`, and DNS) — better for confirming what an app will actually see.

---

### 23. How do you check current active connections and identify a suspicious/high number of connections from one IP?
**Trigger Point:** Suspected DDoS or connection leak.
**Command:**
```bash
ss -tn state established | awk '{print $4}' | cut -d: -f1 | sort | uniq -c | sort -rn | head
netstat -an | grep :80 | awk '{print $5}' | cut -d: -f1 | sort | uniq -c | sort -rn
```
**Flags Explained:**
- `state established` → filters only active (not listening/time-wait) connections.
- Pipeline breakdown: extract remote IP → count occurrences per IP → sort descending — identifies which client IP is opening excessive connections.

---

### 24. How would you check and configure static routes / default gateway on a Linux box?
**Trigger Point:** Multi-NIC server or VPC routing issue.
**Command:**
```bash
ip route show
ip route add 10.0.0.0/24 via 192.168.1.1
ip addr show
```
**Flags Explained:**
- `ip route show` → current routing table.
- `ip route add <subnet> via <gateway>` → adds a static route (non-persistent; for permanence, add to netplan/`ifcfg`/`nmcli` config depending on distro).

## Log Analysis & Text Processing

### 25. How do you find all occurrences of "ERROR" in a log file, showing 5 lines of context around each?
**Trigger Point:** Incident debugging — the most frequently asked log question.
**Command:**
```bash
grep -C 5 "ERROR" app.log
grep -B 5 -A 5 "ERROR" app.log
```
**Flags Explained:**
- `-C 5` → 5 lines of context before AND after each match.
- `-B` = before, `-A` = after (use independently if you need asymmetric context).

---

### 26. How do you tail a log file in real time and filter for a specific pattern?
**Trigger Point:** Live monitoring during a deployment or incident.
**Command:**
```bash
tail -f app.log | grep --line-buffered "ERROR"
tail -n 100 -f app.log
```
**Flags Explained:**
- `-f` → follow (stream new lines as they're written).
- `--line-buffered` → forces `grep` to flush output line-by-line instead of buffering, so matches show up immediately in a piped/streamed context.
- `-n 100` → start by showing the last 100 lines before following.

---

### 27. How do you find how many times each error type occurs in a log file, sorted by frequency?
**Trigger Point:** Tests `awk`/`sort`/`uniq` pipeline skills — very common.
**Command:**
```bash
grep "ERROR" app.log | awk -F'ERROR:' '{print $2}' | sort | uniq -c | sort -rn | head -10
```
**Flags Explained:**
- `awk -F'ERROR:'` → splits each line using "ERROR:" as delimiter, `$2` = text after it (the error message).
- `sort | uniq -c` → `uniq -c` counts consecutive duplicate lines, so input must be sorted first.
- `sort -rn` → sort numerically, descending by count.

---

### 28. How do you extract logs between two timestamps from a large log file?
**Trigger Point:** Incident postmortem — narrowing down a large file to the relevant window.
**Command:**
```bash
sed -n '/2026-07-18 10:00:00/,/2026-07-18 10:15:00/p' app.log
awk '$0 >= "2026-07-18 10:00:00" && $0 <= "2026-07-18 10:15:00"' app.log
```
**Flags Explained:**
- `sed -n '/start/,/end/p'` → `-n` suppresses default output, prints only the range between two pattern matches (inclusive).
- The `awk` version works better if exact timestamp strings aren't guaranteed to appear verbatim in the file.

---

### 29. How do you replace text across multiple files (e.g., updating a config value everywhere)?
**Trigger Point:** Bulk config change across environments.
**Command:**
```bash
sed -i 's/old_value/new_value/g' config.yaml
grep -rl "old_value" /etc/app/ | xargs sed -i 's/old_value/new_value/g'
```
**Flags Explained:**
- `sed -i` → edits file in-place; `s/old/new/g` → substitute, `g` = all occurrences per line (without `g`, only first match per line is replaced).
- `grep -rl` → recursively list filenames (not content) containing the pattern; piped into `xargs sed -i` to bulk-edit all matches. **Best practice**: take a backup first with `sed -i.bak`.

---

### 30. How do you find lines that exist in one file but not another (diffing configs)?
**Trigger Point:** Comparing configs between environments (e.g., UAT vs Prod).
**Command:**
```bash
diff file1.txt file2.txt
comm -23 <(sort file1.txt) <(sort file2.txt)
```
**Flags Explained:**
- `diff` → line-by-line comparison, shows additions/deletions.
- `comm -23` → requires sorted input; suppresses column 2 (lines only in file2) and column 3 (lines common to both), leaving only lines unique to file1.

---

### 31. How do you find the top 10 IPs hitting a web server from an access log?
**Trigger Point:** Nginx/Apache access log analysis — extremely common in interviews.
**Command:**
```bash
awk '{print $1}' access.log | sort | uniq -c | sort -rn | head -10
```
**Flags Explained:**
- `$1` → first field of standard combined log format = client IP.
- Same sort/uniq/sort pattern as Q27 — this combo (`sort | uniq -c | sort -rn`) is the single most reused Linux one-liner pattern in DevOps interviews; know it cold.

---

### 32. How do you search for a pattern recursively across a codebase, excluding certain directories?
**Trigger Point:** Finding a hardcoded secret/config value across a repo.
**Command:**
```bash
grep -rn "pattern" /path --exclude-dir={node_modules,.git,vendor}
grep -rn "pattern" /path --include="*.yaml"
```
**Flags Explained:**
- `-r` → recursive, `-n` → show line numbers.
- `--exclude-dir` → skip specified directories; `--include` → restrict search to matching file patterns.

---

### 33. How do you count the number of lines, words, and characters in a file, and split a large file into chunks?
**Trigger Point:** Handling huge log/data files for processing.
**Command:**
```bash
wc -l file.txt
split -l 100000 bigfile.log chunk_
```
**Flags Explained:**
- `wc -l` → line count (`-w` words, `-c` bytes).
- `split -l 100000` → splits into files of 100,000 lines each, prefixed `chunk_aa`, `chunk_ab`, etc. Useful before feeding large files into tools with memory limits.

## System Monitoring & Performance

### 34. Server load average is high — how do you interpret it and dig deeper?
**Trigger Point:** Very common opener for a troubleshooting round.
**Command:**
```bash
uptime
nproc
top
mpstat 1
```
**Flags Explained:**
- `uptime` load average = avg number of processes waiting for CPU over 1/5/15 min.
- Compare against `nproc` (number of CPU cores) — load of 8 on a 4-core box means real contention; load of 8 on a 32-core box is fine.
- `mpstat 1` → per-CPU core utilization every second, helps spot single-core bottlenecks vs overall load.

---

### 35. How do you check memory usage and understand the difference between used, free, and available memory?
**Trigger Point:** "Free memory shows low" panic — tests real understanding of Linux memory management.
**Command:**
```bash
free -h
cat /proc/meminfo
```
**Flags Explained:**
- `free -h` → human-readable. Key insight: Linux uses free RAM for **disk caching (buff/cache)**, which is reclaimable on demand — it's not "wasted." The **`available`** column is the true free-for-use metric, not the `free` column.
- `/proc/meminfo` → granular breakdown (Cached, Buffers, SwapFree, etc.) for deeper analysis.

---

### 36. How do you check swap usage and whether the system is actively swapping (thrashing)?
**Trigger Point:** App performance degrades under load.
**Command:**
```bash
free -h
vmstat 1 5
swapon --show
```
**Flags Explained:**
- `vmstat` `si`/`so` columns = swap-in/swap-out per second — non-zero and sustained values mean active thrashing, a serious performance red flag.
- `swapon --show` → lists active swap devices/files and usage.

---

### 37. How do you generate a snapshot of system health for a postmortem report?
**Trigger Point:** Post-incident documentation.
**Command:**
```bash
sar -u 1 5      # CPU
sar -r 1 5      # Memory
sar -d 1 5      # Disk
uptime; free -h; df -h; who
```
**Flags Explained:**
- `sar` (System Activity Reporter, from `sysstat` package) → historical + live performance metrics; `-u`=CPU, `-r`=memory, `-d`=disk I/O, `1 5` = 5 samples at 1-second intervals.
- Useful because `sar` can also read historical data logged by `sysstat` from *before* the incident was noticed (`sar -f /var/log/sysstat/saXX`).

---

### 38. How do you check open file descriptor limits, and how do you fix "too many open files" errors?
**Trigger Point:** App crashes under load with `EMFILE`/"too many open files."
**Command:**
```bash
ulimit -n
cat /proc/<PID>/limits
lsof -p <PID> | wc -l
```
**Flags Explained:**
- `ulimit -n` → current soft limit on open file descriptors for the shell/process.
- Fix: raise limits in `/etc/security/limits.conf` (per-user) or in the systemd unit file via `LimitNOFILE=` (for services — `ulimit` set in a shell won't affect systemd-managed services).

---

## Users, Groups & Access

### 39. How do you create a new user with a specific shell, home directory, and add them to a group — without allowing password login?
**Trigger Point:** Onboarding a service account securely.
**Command:**
```bash
useradd -m -d /home/svc_app -s /bin/bash -G docker svc_app
usermod -aG sudo svc_app
passwd -l svc_app
```
**Flags Explained:**
- `-m` → create home directory, `-d` → specify path, `-s` → login shell, `-G` → supplementary groups.
- `usermod -aG` → **append** (`-a`) to group, without `-a` it would overwrite existing group memberships — a very common real-world mistake.
- `passwd -l` → locks the password (disables password-based login; useful for service accounts meant to use SSH keys only).

---

### 40. How do you check which groups a user belongs to, and troubleshoot a "permission denied" tied to group membership?
**Trigger Point:** User added to `docker` group but `docker ps` still fails.
**Command:**
```bash
groups <username>
id <username>
newgrp docker
```
**Flags Explained:**
- `id` → shows UID, GID, and all group memberships in one place.
- Common gotcha: group changes require a **new login session** to take effect; `newgrp docker` starts a subshell with the new group active immediately without re-login.

---

### 41. How do you audit sudo access and see what commands a user has run with sudo?
**Trigger Point:** Security audit / compliance question.
**Command:**
```bash
cat /etc/sudoers
sudo -l -U <username>
grep sudo /var/log/auth.log      # Debian/Ubuntu
grep sudo /var/log/secure        # RHEL/CentOS
```
**Flags Explained:**
- `sudo -l -U` → lists the sudo privileges a specific user has, without needing to read the raw sudoers file.
- Always edit sudoers with `visudo` (never directly) — it validates syntax before saving, preventing a lockout from a typo.

## Services & systemd

### 42. A service fails to start after deployment. How do you investigate?
**Trigger Point:** Extremely common — tests the full systemd debugging workflow.
**Command:**
```bash
systemctl status myapp.service
journalctl -u myapp.service -n 100 --no-pager
journalctl -u myapp.service --since "10 minutes ago"
systemctl show myapp.service
```
**Flags Explained:**
- `systemctl status` → quick state + last few log lines + exit code.
- `journalctl -u` → full logs for that specific unit; `-n 100` → last 100 lines; `--since` → time-filtered logs; `--no-pager` → print directly instead of opening `less`.
- `systemctl show` → dumps every property of the unit (Environment vars, ExecStart, dependencies) — useful when the issue is a misconfigured unit file.

---

### 43. How do you make a custom application run as a systemd service, auto-restart on failure, and start on boot?
**Trigger Point:** Deploying a custom app/script as a managed service — asked constantly.
**Command:**
```ini
# /etc/systemd/system/myapp.service
[Unit]
Description=My App
After=network.target

[Service]
ExecStart=/usr/bin/python3 /opt/app/main.py
Restart=on-failure
RestartSec=5
User=appuser

[Install]
WantedBy=multi-user.target
```
```bash
systemctl daemon-reload
systemctl enable --now myapp.service
```
**Flags Explained:**
- `After=network.target` → ensures the service starts only after networking is up.
- `Restart=on-failure` → auto-restarts only on non-zero exit (not on clean stop); `RestartSec=5` → wait 5s before retry.
- `daemon-reload` → required after any unit file change, so systemd re-reads it.
- `enable --now` → enables on boot AND starts it immediately in one command.

---

### 44. How do you check which services are enabled at boot, and disable one that's unnecessary?
**Trigger Point:** Hardening / startup-time optimization.
**Command:**
```bash
systemctl list-unit-files --state=enabled
systemctl disable <service>
systemctl mask <service>
```
**Flags Explained:**
- `list-unit-files --state=enabled` → filters to only services set to auto-start.
- `disable` → stops it from starting at boot (but can still be started manually).
- `mask` → stronger — symlinks the unit to `/dev/null`, so it can't be started even manually/accidentally by another unit's dependency.

### 45. A service is in a restart loop. How do you diagnose it without letting it keep crash-looping?
**Trigger Point:** Real production firefighting scenario.
**Command:**
```bash
systemctl stop myapp.service
journalctl -u myapp.service -n 200
systemctl status myapp.service --no-pager -l
```
**Flags Explained:**
- Stop it first to break the crash loop and prevent resource exhaustion/log flooding, THEN inspect the last known logs (`journalctl` retains logs even after stop).
- `-l` (`--full`) in `status` → don't truncate long lines, important for full stack traces.

## Archiving, Compression & File Transfer

### 48. How do you create and extract a `.tar.gz` archive, and check its contents without extracting?
**Trigger Point:** Basic but always tested — backup/restore workflows.
**Command:**
```bash
tar -czvf backup.tar.gz /path/to/dir
tar -xzvf backup.tar.gz -C /restore/path
tar -tzvf backup.tar.gz
```
**Flags Explained:**
- `c` = create, `x` = extract, `t` = list/table of contents (no extraction), `z` = gzip compression, `v` = verbose, `f` = specify filename (must come last before the filename argument).
- `-C` → change to target directory before extracting (avoids extracting into the wrong place).

---

### 49. How do you copy files between servers securely, and resume an interrupted large transfer?
**Trigger Point:** Deploying large artifacts, or backup/restore across hosts.
**Command:**
```bash
scp -r /local/dir user@remote:/remote/dir
rsync -avz --progress /local/dir/ user@remote:/remote/dir/
rsync -avz --partial --progress source/ dest/
```
**Flags Explained:**
- `scp -r` → recursive copy over SSH; simple but restarts fully on failure.
- `rsync -a` → archive mode (preserves permissions, symlinks, timestamps); `-v` verbose; `-z` compress in transit; `--partial` → keeps partially-transferred files so a retry resumes instead of restarting.
- Trailing slash on source (`dir/`) matters: copies *contents* of dir, not the dir itself.

---

### 50. How do you find and delete files older than N days (log rotation / cleanup scenario)?
**Trigger Point:** Extremely common scripting question — often tied to `/var/log` cleanup or old backups.
**Command:**
```bash
find /var/log/app -type f -name "*.log" -mtime +7 -exec rm -f {} \;
find /backups -type f -mtime +30 -print -delete
```
**Flags Explained:**
- `-mtime +7` → modified more than 7 days ago (`+` = older than, `-` = newer than).
- `-exec rm -f {} \;` → runs `rm -f` on each match; `{}` is the placeholder, `\;` terminates the `-exec` clause.
- `-delete` → built-in `find` flag, faster than `-exec rm` since it avoids spawning a new process per file — but must come after all filter conditions, and doesn't work well combined with `-exec` in the same command.

## Cron, Scheduling & Automation

### 51. A cron job isn't running as expected even though it works when run manually. Why, and how do you debug it?
**Trigger Point:** One of the most classic "gotcha" interview scenarios.
**Command:**
```bash
crontab -l
grep CRON /var/log/syslog     # or /var/log/cron on RHEL
env -i /bin/bash --noprofile --norc -c "/path/to/script.sh"
```
**Flags Explained:**
- Root cause almost always: **cron runs with a minimal environment** — no `$PATH`, no user profile/env vars loaded. Scripts using relative paths or assuming a full shell environment fail silently.
- Fix: use absolute paths in the script, explicitly `source` needed env files, or set `PATH=` at the top of the crontab.
- `env -i ... bash --noprofile --norc` → simulates cron's bare environment locally to reproduce the failure before deploying a fix.
- `grep CRON /var/log/syslog` → confirms whether cron even attempted to trigger the job.

---

### 52. Explain crontab syntax and how you'd schedule a job to run every 15 minutes during business hours on weekdays.
**Trigger Point:** Tests actual cron syntax fluency, not just "I use cron."
**Command:**
```bash
*/15 9-17 * * 1-5 /opt/scripts/healthcheck.sh
```
**Flags Explained:**
- Fields in order: `minute hour day-of-month month day-of-week`.
- `*/15` → every 15 minutes; `9-17` → 9 AM–5 PM; `1-5` → Monday–Friday (0=Sunday).
- Always redirect output in production cron entries: `... >> /var/log/healthcheck.log 2>&1` to avoid silent failures and mail spam.

---

### 53. How do you schedule a one-time task to run later without using cron?
**Trigger Point:** Tests knowledge beyond cron — `at`/`systemd-run`.
**Command:**
```bash
echo "/opt/scripts/cleanup.sh" | at now + 1 hour
atq
systemd-run --on-active=1h /opt/scripts/cleanup.sh
```
**Flags Explained:**
- `at` → schedules a single future execution (requires the `at` package/daemon running).
- `atq` → lists pending `at` jobs.
- `systemd-run --on-active` → modern alternative, creates a transient systemd timer unit for one-off scheduling.

## Boot Process & Troubleshooting

### 54. A server fails to boot / drops into emergency mode. How do you diagnose it?
**Trigger Point:** High-stakes SRE-style scenario question.
**Command:**
```bash
journalctl -xb
systemctl --failed
dmesg | tail -50
```
**Flags Explained:**
- `journalctl -xb` → boot logs for the current (or `-b -1` for previous) boot, `-x` adds explanatory help text for errors.
- `systemctl --failed` → lists all units that failed to start — usually pinpoints the exact blocker (often a misconfigured fstab entry or a failed mount).
- Classic real cause: a bad entry in `/etc/fstab` (e.g., an unreachable NFS mount) blocking boot — fixed by editing fstab in single-user/rescue mode and adding the `nofail` option.

---

### 55. How do you check system boot time and identify what's slowing down startup?
**Trigger Point:** Performance tuning / VM image optimization question.
**Command:**
```bash
systemd-analyze
systemd-analyze blame
systemd-analyze critical-chain
```
**Flags Explained:**
- `systemd-analyze` → total boot time breakdown (kernel vs userspace).
- `blame` → lists units sorted by how long each took to initialize — quickly surfaces the slowest service.
- `critical-chain` → shows the dependency chain that determined total boot time (useful when the delay is caused by a chain of `After=` dependencies, not a single slow unit).

## Security & SSH

### 56. How do you troubleshoot SSH connection failures ("Connection refused" or "Permission denied (publickey)")?
**Trigger Point:** Extremely common — access issues are a daily DevOps reality.
**Command:**
```bash
ssh -vvv user@host
systemctl status sshd
tail -f /var/log/auth.log     # Debian/Ubuntu
tail -f /var/log/secure       # RHEL/CentOS
ls -la ~/.ssh
```
**Flags Explained:**
- `-vvv` → maximum verbosity on the client side, shows exactly which auth methods were tried and why they failed.
- "Connection refused" = network/firewall/sshd-down issue (server-side, before auth). "Permission denied (publickey)" = the SSH daemon is reachable but key-based auth failed — check `~/.ssh/authorized_keys` permissions (must be `600`) and `~/.ssh` directory permissions (must be `700`) — sshd silently rejects keys if permissions are too open.

---

### 57. How do you restrict SSH access to key-based auth only and disable root login?
**Trigger Point:** Server hardening — standard security checklist item.
**Command:**
```bash
# /etc/ssh/sshd_config
PasswordAuthentication no
PermitRootLogin no
```
```bash
sshd -t
systemctl restart sshd
```
**Flags Explained:**
- `sshd -t` → **tests config syntax before restarting** — critical step, since a typo here can lock you out entirely if you restart blindly (especially on a remote-only server with no console access).
- Always keep an existing SSH session open while testing changes, in case the new config breaks connectivity.

---

### 58. How do you check for failed login attempts and block an IP showing brute-force behavior?
**Trigger Point:** Security incident response.
**Command:**
```bash
grep "Failed password" /var/log/auth.log | awk '{print $(NF-3)}' | sort | uniq -c | sort -rn | head
iptables -A INPUT -s <malicious_IP> -j DROP
fail2ban-client status sshd
```
**Flags Explained:**
- Extracts and counts failed-login source IPs using the same `sort | uniq -c | sort -rn` pattern from earlier.
- `iptables -A INPUT -s <IP> -j DROP` → appends a rule dropping all traffic from that IP.
- `fail2ban-client` → checks status of the `fail2ban` service, which automates this IP-banning process based on log pattern matching (preferred over manual iptables rules in production).

---

## Quick-Reference: The Core Pattern to Remember

Almost 30% of scenario questions reduce to this exact pipeline — know it cold:
```bash
<extract field> | sort | uniq -c | sort -rn | head
```
Used for: top IPs, top errors, top users, top processes by frequency — across logs, auth files, and access logs.
